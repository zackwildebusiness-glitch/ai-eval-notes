# AI Failure-Mode Notes

Short write-ups of real failures I've hit while building with and evaluating LLMs and AI
coding agents — analyzed the way an evaluator would: the **failure mode**, the **signal**
that catches it, and the **mitigation**.

My working premise across all of my projects is one sentence: **I treat AI output as
untrusted input.** I build the validation, bounding, fallback, and audit layers around
unreliable models — and I keep notes on where they fail, because noticing and categorizing
failure is the core of evaluation work.

These are real incidents from my own work, not hypotheticals.

---

## 1. Over-literal execution under ambiguous scope

**What happened.** I told an LLM coding agent to "put my projects on GitHub." It read the
vague instruction as authorization to push entire folders — including confidential files.

**Failure mode.** Over-literal instruction-following under ambiguous scope: the model
optimized the literal command and ignored the user's unstated but reasonably inferable
privacy boundary.

**Signal.** The right question isn't "did the git push succeed" — it's *"did the agent
preserve the inferable privacy boundary under ambiguity?"* An eval that only checks task
completion misses this entirely.

**Mitigation.** Split the action into **discovery → prepare → review → publish**, with the
publish step gated behind an exact file manifest, a secret scan, and explicit approval.
Never let an ambiguous instruction authorize an irreversible, broad action.

---

## 2. Unreliable self-report ("I'm asleep" — while working)

**What happened.** An agent-monitoring tool I built reported sub-agents as idle while they
were actively generating output. The status came from each agent's own signal, not from
evidence.

**Failure mode.** Trusting an agent's self-attested state. Self-report is a *claim*, and a
confident wrong claim is worse than no claim.

**Signal.** Compare the self-reported state against ground truth (transcript write activity,
live process markers). Divergence between "what the agent says it's doing" and "what it's
observably doing" is the finding.

**Mitigation.** Treat self-report as a claim to verify, never a fact. I rebuilt the monitor
to derive status from ground truth and flag disagreement, rather than echo the agent.

---

## 3. Hallucinated metric ("the 216 tests that weren't")

**What happened.** A test count of "216" propagated across a résumé and a portfolio from an
unverified doc claim. The code actually held ~150. An AI-assisted summary had carried the
number forward without checking it against source.

**Failure mode.** Confident, specific, wrong figures — the easiest kind of overclaim to
falsify, and the most damaging, because a reviewer who clones the repo and runs the tests
now distrusts every other number.

**Signal.** Any quantitative claim that doesn't trace to a source count is suspect. The
tell is specificity without provenance.

**Mitigation.** Every printed metric must trace to the command that produced it, verified
before publishing. Numbers are generated by running the check, not recalled from memory.
